Propósito Fundamental

Herramienta de diagnóstico hardware/software para análisis de laptops que no encienden o no dan vídeo, permitiendo la caracterización eléctrica y protocolar de sus subsistemas críticos sin depender del funcionamiento del equipo bajo prueba.

Firmware
  • ✓ RP2040 (Arduino IDE 2.3.x)
  • ✓ Core: earlephilhower-rp2040
  • ✓ RTOS: FreeRTOS con tareas priorizadas
  • ✓ Límite: <500 líneas en .ino
Host
  • ✓ Python (PyQt)
  • ✓ UI y procesamiento pesado
  • ✓ Logging y visualización
  • ✓ Comunicación CDC-USB

Restricciones de Diseño (Hard Constraints)
1. Límite de Líneas en .ino

.ino principal < 500 líneas. No agregar lógica compleja ahí. Toda la lógica se segmenta en módulos .cpp/.h separados.

2. No Romper Funcionalidad Validada

Si un módulo ya corrige un error, se descarta la comprobación periódica o el parche redundante. Priorizar estabilidad.

3. Headers PIO .h

Se generan externamente y se pegan tal cual. No intentar regenerarlos o modificar su estructura interna.

4. FreeRTOS

Tareas deben ser no bloqueantes. Usar colas para paso de datos entre adquisición y comunicación.

5. Bus I2C Compartido

El diagnóstico SMBus y el módulo PD comparten físicamente el bus. Cualquier cambio debe considerar multiplexión o exclusión mutua para evitar colisiones.

Flujo Operativo
1PC → Comando START
2[RP2040] → Inicia tareas de adquisición cíclica
3[Sensores] → Lectura INA219/ADS1115/GPIO/PIO
4[UART_EC] → Captura pasiva de tramas EC (si hay actividad)
5[SMBus] → Escaneo/lectura de registros laptop (opcional)
6[RP2040] → Empaqueta datos en estructura JSON-like por UART
7[PC] → Parsea, actualiza UI, almacena logs
8[PC] → Permite inyección de comandos para pruebas activas
Instrucciones para IAs Receptoras
  • Contexto técnico fijo: Cualquier sugerencia, refactor o generación de código debe respetar la arquitectura modular, el límite de líneas en .ino, la versión del core, y el flujo FreeRTOS.
  • Priorizar estabilidad: Claridad y compatibilidad. No sugerir reemplazos de módulos ya validados.
  • Código nuevo: Si se propone código nuevo, entregar solo el fragmento afectado con comentarios de integración. Mantener compatibilidad con Arduino IDE 2.3.x y la estructura de cola/comandos existente.
  • Procesamiento pesado: La UI y procesamiento residen siempre en el host PC. El firmware solo adquiere, empaqueta y obedece comandos.
